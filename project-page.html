The Problem

Agent Zero is powerful.
But it wasn't built for work.

Agent Zero is an excellent open-source autonomous agent framework — flexible, multi-model, capable of long-horizon tasks. But its defaults are oriented toward individual developers hacking on their home machine: no authentication hardening, arbitrary code execution over HTTP, phone-home update checks, hacker-themed UI, and cost-inefficient model settings.

Sentinel is my answer to that gap. It takes Agent Zero as a foundation and layers exactly what's needed to run it safely, reliably, and cheaply in a corporate engineering environment — without diverging so far from upstream that merging improvements becomes painful.

Core Framework

Multi-agent architecture

Hierarchical agent delegation — a primary orchestrator spins up specialised sub-agents for development, research, or custom tasks. Each agent runs with its own context, memory, and model configuration.

Real-time WebSocket communication

All agent activity streams to the browser over Socket.IO — progress, tool calls, sub-agent spawning, and final output — no polling required. Full duplex, with origin validation and connection lifecycle management.

Project workspaces

Isolated project contexts with their own memory, file space, and history. Switch between projects from the sidebar. Each workspace persists independently so context doesn't bleed between tasks.

Memory system

Semantic vector memory backed by FAISS and sentence-transformers. Agents can save, recall, and forget information across sessions. Memory is scoped per-project and per-agent.

Skill system with approval workflow

Agents can acquire new skills (composable tool definitions). Imported skills land in a pending/ queue and require explicit human promotion before execution — a critical safety gate.

Task scheduler

Cron-style task scheduling with a full UI — create, view, enable, and delete scheduled agent tasks. Built on top of the extension hook system for clean separation of concerns.

Web UI

Chat interface

Streaming chat with the active agent. Supports markdown, code blocks with syntax highlighting, KaTeX math rendering, image attachments, and collapsible sub-agent process trees.

File manager

In-browser file browser and editor (Ace Editor) for the agent's working directory. Drag-and-drop upload, download, and delete. Full path navigation with breadcrumbs.

Settings UI

Configure model providers, API keys, network/TLS settings, MCP servers, and agent profiles — all from the browser, no config file editing required.

Scheduler UI

Visual task scheduler with cron expression input, enable/disable toggles, and last-run status. Persists across container restarts.

Memory dashboard

Browse, search, and delete stored memories. Shows memory source, timestamp, and relevance. Supports bulk operations for housekeeping.

PWA support

Full Progressive Web App manifest with service worker for offline caching. Installable on desktop and mobile. Responds to system dark/light mode preferences.

Agent Capabilities

Code execution sandbox

Python, Node.js, and shell execution inside the Docker container. Module allowlist enforced. The /rfc arbitrary-execution endpoint is disabled in production mode.

Browser automation

Full Playwright-powered browser agent for web research, form filling, and scraping. Downloads disabled, security constraints on, proxy-aware, with prompt injection defence on web content.

Document processing

PDF ingestion (PyPDF, PyMuPDF, PDF2Image), OCR, HTML parsing (BeautifulSoup), and article extraction (Newspaper3k). Feeds the agent's knowledge base and working memory.

Voice interface

Speech-to-text (Whisper) and text-to-speech (Kokoro TTS) for hands-free operation. Configurable via Settings. Useful for ambient monitoring tasks during long operations.

Email integration

SMTP/IMAP and Exchange/Outlook support via exchangelib. Credentials stored in the encrypted secrets manager. Agents can read, draft, and send email as part of workflows.

SSH terminal access

Paramiko-based SSH client for agents to operate on remote systems. Useful for deployment pipelines, server administration, and infrastructure automation tasks.

Security hardening

Production security,
not afterthought security.

Security hardening was the first priority — not something to bolt on later. Before any feature work, Sentinel addressed the most significant risks present in a default Agent Zero deployment.

Authentication & Sessions

  • ✓ SHA256 credential hashing (not plaintext comparison)
  • ✓ 1-second enforced delay on failed login (brute-force mitigation)
  • ✓ Session cookies with SameSite=Strict
  • ✓ CSRF dual-layer: session token + runtime-scoped secrets.token_urlsafe(32) cookie
  • ✓ Login page does not leak whether username or password was wrong

Network & API

  • ✓ WebSocket origin validation (RFC 6455 compliant)
  • ✓ DNS rebinding prevention via origin allowlist (ALLOWED_ORIGINS)
  • ✓ Sensitive endpoints restricted to loopback via requires_loopback()
  • ✓ API key auth via requires_api_key() for MCP/external endpoints
  • ✓ Container port bound to 127.0.0.1 only — not all interfaces
  • ✓ nginx server_tokens off — version hidden from responses

Code Execution Controls

  • ✓ /rfc endpoint (arbitrary Python execution over HTTP) disabled in Docker/production
  • ✓ Strict module allowlist enforced in dev mode (python.helpers.*, python.api.*, python.tools.*)
  • ✓ Skill approval workflow: new skills must be promoted from pending/ by a human

Input & File Safety

  • ✓ Unicode normalization on all filenames
  • ✓ Forbidden character blocking (null bytes, control chars, path separators)
  • ✓ Windows reserved name detection
  • ✓ Path traversal prevention (../ sequences stripped)
  • ✓ 255-byte filename limit enforced
  • ✓ Browser web content treated as untrusted (prompt injection defence)

🚫
Update checker removed

Upstream Agent Zero phones home to api.agent-zero.ai on every user message to check for updates. This call was removed entirely in Sentinel — no telemetry, no external dependency at runtime, no data leakage to third-party servers in a corporate deployment.

Current state

Where things stand today.

As of March 2026, Sentinel is actively maintained and fully deployable. Phase 1 (security hardening) is largely complete. Phase 2 and 3 are on the roadmap but not yet started.

Base image swap Done

Swapped the Docker base image from Kali Linux to Ubuntu 24.04. The Kali base was a remnant of a planned "hacker agent" profile — removed entirely. Ubuntu is leaner, more appropriate for a corporate deployment, and has a better security posture out of the box.

Security vulnerability sweep Done

Audited all 62 Python dependencies and Node.js packages. Bumped versions to resolve 25 CVEs across pypdf, langchain-core, flask, mcp, unstructured, lxml_html_clean, markdown, and others. Two remain blocked by upstream release schedules.

Authentication & CSRF hardening Done

SHA256 credential hashing, brute-force delay, SameSite=Strict session cookies, and dual-layer CSRF protection. The login flow no longer leaks whether the username or password was incorrect.

WebSocket & network hardening Done

RFC 6455-compliant WebSocket origin validation, DNS rebinding prevention, loopback-only container port binding, nginx server token suppression, and requires_loopback() / requires_api_key() endpoint decorators.

TLS / SSL management Done

Centralised TLS configuration via environment variables. Custom CA bundle upload via Settings UI — supports .pem, .crt, .cer, .ca-bundle. Settings propagated to all HTTP clients: aiohttp, httpx, requests, Playwright, IMAPClient.

Per-agent model configuration Done

Each agent profile can now override the global model selection. Configured via the Settings UI. Allows Developer to run on a capable coding model while the Researcher uses a cheaper one for document work — without touching config files.

Build pipeline Done

Production-grade build and run scripts (build.sh, run.sh, stop.sh, test.sh). Date-stamped image tagging (YYYYMMDD_HH_MM). Multi-instance support via environment variable namespacing. Test instance runs on isolated port and volume alongside production.

Audit logging Planned

Structured logging of who ran what, when — essential for enterprise compliance. Architecture is sketched: extend the existing extension hook system so every tool call emits a signed log entry. Not yet implemented.

Full API authentication audit Planned

A systematic pass through all 77 API endpoints to verify correct authentication requirements. Some endpoints added by upstream may not have inherited the requires_api_key() decorator. Needs a dedicated audit pass.

Design philosophy

Extend, don't diverge.

The hardest part of forking an active project isn't the initial changes — it's staying mergeable as upstream evolves.

Staying upstream-compatible

Every Sentinel-specific change in Python source is annotated with a # SENTINEL: comment. This makes it straightforward to review a new upstream commit and determine whether it conflicts with Sentinel's modifications — and to cherry-pick improvements without a full rebase.

The goal is to never be so far from upstream that merging becomes a multi-day project. The changes Sentinel makes are mostly additive or restrictive (disabling things, adding checks) rather than architectural rewrites.

What Sentinel is not

Sentinel is not trying to replace Agent Zero or build a competing product. It's a personal project to make Agent Zero practical for one specific use case: self-hosted deployment in a corporate engineering environment where you need to trust the system you're running.

It's also not a SaaS product, a managed service, or something with a commercial angle. It's a tool I built and maintain for my own use, open-sourced so others in the same situation can benefit or contribute.
